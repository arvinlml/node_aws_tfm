# Branch Protection Rules Configuration Guide

This repository implements branch protection rules to enforce code quality, security, and collaboration standards. Rules are configured through code/configuration for consistency and auditability.

## Overview

Branch protection rules are configured in two ways:

1. **Declarative Configuration** (`.github/settings.yml`) - Using Probot Settings App
2. **GitHub Actions Workflow** (`configure-branch-protection.yml`) - Automated setup
3. **Shell Scripts** (in `.github/scripts/`) - Manual management

## Protected Branches

### `main` Branch

- **Production/Release Branch**
- Requires: 1 approved review + all status checks
- Status checks: test (18.x, 20.x), lint, type-check, **coverage**
- Dismisses stale reviews: Yes
- Requires branch to be up to date: Yes
- No force pushes or deletions allowed

### `develop` Branch

- **Development/Integration Branch**
- Requires: 1 approved review + status checks
- Status checks: test (18.x, 20.x), lint, type-check
- Dismisses stale reviews: Yes
- Requires branch to be up to date: Yes
- No force pushes or deletions allowed

## Setup Instructions

### Option 1: Using Probot Settings App (Recommended)

The most elegant solution using `.github/settings.yml`:

1. **Install the Settings App**
   - Go to: https://github.com/apps/settings
   - Click "Install"
   - Select your repository

2. **Settings will be applied automatically**
   - The app watches `.github/settings.yml`
   - Changes to the file trigger automatic rule updates
   - All configuration is version-controlled

3. **Benefits**
   - Fully declarative and version-controlled
   - No manual GitHub UI configuration needed
   - Changes tracked in git history
   - Applies to all cloned repositories

### Option 2: Using GitHub Actions Workflow

Automatic setup via GitHub Actions:

1. **Trigger the workflow**

   ```bash
   gh workflow run configure-branch-protection.yml -r main
   ```

2. **Or manually trigger**
   - Go to "Actions" tab
   - Select "Configure Branch Protection" workflow
   - Click "Run workflow"

3. **Benefits**
   - No external app required
   - Triggered on changes to the configuration
   - Auditable via Actions logs

### Option 3: Using Shell Scripts (Manual)

Using the provided helper scripts:

#### Prerequisites

```bash
# Install GitHub CLI
brew install gh

# Authenticate
gh auth login
```

#### Setup Branch Protection

```bash
# Setup both main and develop
bash .github/scripts/setup-branch-protection.sh

# Setup only main
bash .github/scripts/setup-branch-protection.sh --main

# Setup only develop
bash .github/scripts/setup-branch-protection.sh --develop
```

#### View Current Rules

```bash
# View all protected branches
bash .github/scripts/view-branch-protection.sh

# View specific branch
bash .github/scripts/view-branch-protection.sh main
bash .github/scripts/view-branch-protection.sh develop
```

#### Remove Protection

```bash
# Remove protection from develop (use with caution!)
bash .github/scripts/remove-branch-protection.sh develop
```

## Configuration Details

### Required Status Checks

The workflows must define these status check names:

#### Main Branch (`main`)

```
test (18.x)
test (20.x)
lint
type-check
coverage
```

#### Develop Branch (`develop`)

```
test (18.x)
test (20.x)
lint
type-check
```

These are generated by `.github/workflows/ci.yml`:

- Test jobs for Node 18.x and 20.x
- Linting checks
- Type checking
- Coverage reporting

### Pull Request Requirements

#### Before Merge

- ✅ At least 1 approval required
- ✅ All status checks must pass
- ✅ Branch must be up to date with base branch
- ✅ Stale reviews are dismissed when new commits pushed

#### Restrictions

- ❌ Force pushes not allowed
- ❌ Direct pushes to protected branches not allowed
- ❌ Branch deletions not allowed

## Updating Protection Rules

### Modifying Rules

1. **Using Probot Settings App**

   ```bash
   # Edit .github/settings.yml
   git add .github/settings.yml
   git commit -m "docs: update branch protection rules"
   git push
   # App automatically applies changes
   ```

2. **Using GitHub Actions**

   ```bash
   # Update configure-branch-protection.yml
   git add .github/workflows/configure-branch-protection.yml
   git commit -m "ci: update branch protection configuration"
   git push
   # Workflow applies changes automatically
   ```

3. **Using Scripts**
   ```bash
   # Edit .github/scripts/setup-branch-protection.sh
   bash .github/scripts/setup-branch-protection.sh
   ```

## Troubleshooting

### Rules Not Applied

```bash
# Check if Probot Settings app is installed
gh api repos/{owner}/{repo}/installations

# Verify configuration syntax
git diff .github/settings.yml
```

### Missing Status Checks

```bash
# Status checks must be defined in CI workflow
# Check workflows/ci.yml for all job names

# View current branch protection
bash .github/scripts/view-branch-protection.sh main
```

### Cannot Push to Protected Branch

- Create a pull request instead
- Get 1 approval
- All status checks must pass
- Branch must be up to date

### Re-apply Rules After Removal

```bash
bash .github/scripts/setup-branch-protection.sh
```

## Permissions Required

- **Repository**: Owner/Admin role
- **GitHub CLI**: Authenticated with appropriate scopes
- **Probot Settings App**: Repository access granted
- **GitHub Actions**: Default GITHUB_TOKEN with necessary permissions

## Best Practices

### Code Review

- ✅ Require at least 1 review for all changes
- ✅ Dismiss stale reviews to ensure current context
- ✅ Use code owners file for automatic reviewers

### Testing

- ✅ Run tests on all supported Node versions
- ✅ Require linting checks
- ✅ Require type checking
- ✅ Enforce minimum coverage thresholds

### Merge Strategy

- ✅ Delete branches after merge (configured in settings.yml)
- ✅ Use squash merge for cleaner history
- ✅ Require PR number in commit messages

### Documentation

- ✅ Keep rules version-controlled
- ✅ Document changes in git history
- ✅ Update this guide when rules change

## Example Workflow

### Making Changes

1. **Create feature branch**

   ```bash
   git checkout -b feature/new-feature
   ```

2. **Make changes**

   ```bash
   # Code changes
   npm run lint:fix
   npm test
   ```

3. **Push and create PR**

   ```bash
   git push origin feature/new-feature
   # Open PR on GitHub
   ```

4. **Automated checks**
   - CI workflow runs automatically
   - Linting, testing, type-checking
   - Coverage report posted

5. **Review and merge**
   - Gets 1+ approvals
   - All checks pass
   - Merge button enabled
   - Branch deleted automatically
